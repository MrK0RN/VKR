# План реализации: дифференциальная диагностика лимфомы Ходжкина

**Заболевание:** лимфома Ходжкина (две формы)  
**Исходная модель:** `Сеть Петри Лимфома Ходжкина.docx`  
**Стек:** Python 3.11+, FastAPI, Jinja2/HTML, session-based storage  
**Принцип:** программная интерпретация сети Петри (не if-else дерево), логика модели не меняется

---

## 1. Цели и границы проекта

| Параметр | Значение |
|----------|----------|
| Форма 1 (выход) | Нодулярный склероз — позиция **b52** |
| Форма 2 (выход) | Смешанно-клеточный склероз — позиция **b53** |
| Разделы опросника | Жалобы → Осмотр → Лабораторные → Инструментальные |
| Промежуточные позиции | b9/b10, b11/b12, b26/b27, b42/b43 |
| Минимальный порог диагноза | **Нет** — показывать баллы как есть, в т.ч. 0/0 |
| Числовые признаки | Чекбокс «Показатель превышает порог: да/нет» (без ввода числа) |
| История | Сохранение по **сессии** (cookie/session id), без долговременной БД пациентов на первом этапе |

## 2. Модель сети Петри (каноническое описание)

### 2.1. Структура

Сеть — **ориентированный двудольный граф**:

- **Places (позиции):** клинические признаки `b1…b51`, промежуточные `b9, b10, b11, b12, b26, b27, b42, b43`, финальные `b52, b53`.
- **Transitions (переходы):** срабатывание при накоплении маркеров/весов на входных дугах; не дублировать в коде как «if symptom then add 2» вне движка.
- **Tokens / marking:** отмеченный признак = маркер в соответствующей позиции; веса на дугах к формам b52/b53 накапливаются в **маркировке счётчиков** `score_nodular`, `score_mixed`.

### 2.2. Веса

Формат в таблице: `(X)` — только нодулярный; пусто во второй колонке — только смешанно-клеточный; `(X) / (Y)` — при отметке признака **+X** к b52-ветке и **+Y** к b53-ветке.

Примеры общих признаков: b4–b6, b18–b20, b31–b34 — двойные веса.

### 2.3. Промежуточные позиции (вариант А)

Промежуточные места **не блокируют** переход к следующему разделу как жёсткий порог. Они:

1. Отражают **достижимость** ветки (есть ли маркеры, ведущие к b9 vs b10 после жалоб и т.д.).
2. При **невыполнении** условий перехода показывают **информационное окно** (не ошибка валидации):  
   *«Не выявлено подозрений на нодулярную / смешанно-клеточную форму на данном этапе»* — пользователь может продолжить.

Условия перехода для промежуточных мест задать в конфиге сети (см. §4), согласованно с docx.

### 2.4. Особое правило последнего блока (инструментальные)

Из примечания к модели:

> Переход между b42 и b52, b43 и b53 возможен при выполнении **хотя бы одного** b из последнего блока (b44–b51).

**Интерпретация для приложения:**

- На этапах 1–3 для «срабатывания» перехода к промежуточной позиции следующего уровня учитывается **накопленная сумма весов** по соответствующей ветке (как в сети).
- На этапе 4 (**инструментальные**): для финального предсказания достаточно **≥1 отмеченного** признака b44–b51 с любым весом; не требуется несколько признаков.
- Если инструментальный блок пустой — финальный расчёт всё равно выполняется по накопленным весам из предыдущих разделов (0 по инструментальным), переход b42→b52 / b43→b53 по инструментальным не «срабатывает», но итоговые баллы показываются.

### 2.5. Итоговый результат

- `score_nodular` — сумма весов по ветке к b52.
- `score_mixed` — сумма весов по ветке к b53.
- Наиболее вероятная форма: `argmax` (при равенстве — явно указать «формы равновероятны по баллам»).
- Опционально: список признаков, внесших вклад (топ по весу).

## 5. Движок сети Петри (`petri/`)

### 5.1. Классы

| Класс | Ответственность |
|-------|----------------|
| `Place` | id, label, section, marked |
| `Arc` | source_place, target (place или `score_*`), weight |
| `Transition` | id, guard (callable из конфига), input/output places |
| `Marking` | `places: dict[str, int]`, `scores: dict[str, float]` |
| `PetriNet` | граф, загрузка из JSON |
| `PetriEngine` | `apply_user_input(place_id, value)`, `fire_transition(t_id)`, `get_marking()` |

### 5.2. Алгоритм на шаге опросника

1. Пользователь отправляет форму раздела → для каждого `place` с `value=true` вызывается `engine.mark(place_id)`.
2. По исходящим дугам увеличиваются `scores` (не отдельные переменные в роутере).
3. Вызывается `engine.evaluate_section_transition(section)`:
   - проверка guard перехода (сумма / any_one для последнего блока);
   - установка токенов в b9/b10/… при выполнении;
   - возврат `TransitionResult { fired, warning_message? }`.
4. Состояние сериализуется в сессию.

### 5.3. Финальный расчёт

`engine.finalize()` → читает `scores`, определяет победителя, `explain()` обходит отмеченные places с ненулевым вкладом.

### 5.4. Тесты

- Сценарий «только нодулярные жалобы» → b9 срабатывает, score_nodular > 0.
- Сценарий «все пусто» → 0/0, без исключений.
- Сценарий «один b44» → переход instrumental→final по правилу any_one.
- Регрессия: сумма весов b4 (1)/(2) = +1 nodular, +2 mixed.

## 6. Веб-интерфейс

### 6.1. Поток

1. Старт → создаётся session_id.
2. Шаги 1–4: форма с чекбоксами; для пороговых — подпись порога из модели + «Да / Нет».
3. Кнопки **Назад** / **Далее** — восстановление отмеченных значений из сессии.
4. После шага — при `warning_message` модальное/alert окно, затем переход дальше.
5. Страница **Результат**: форма, баллы, опционально таблица вкладов признаков.

### 6.2. API (пример)

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/` | редирект на шаг 1 |
| GET | `/step/{n}` | форма раздела n |
| POST | `/step/{n}` | сохранить ответы, evaluate transition |
| GET | `/result` | итог |
| POST | `/reset` | новая сессия |

## 7. Логирование

### 7.1. Требования

- Структурированные логи (JSON в файл + человекочитаемый консоль при dev).
- **Не логировать** персональные данные пациента (если позже появятся ФИО) — только session_id и технические поля.

### 7.2. События

| Уровень | Событие | Поля |
|---------|---------|------|
| INFO | `session_started` | session_id |
| DEBUG | `place_marked` | session_id, place_id, weights_applied |
| INFO | `transition_evaluated` | session_id, transition_id, fired, scores |
| WARNING | `transition_warning_shown` | session_id, message |
| INFO | `diagnosis_completed` | session_id, score_nodular, score_mixed, winner |
| ERROR | `engine_error` | session_id, exception, stack |

### 7.3. Реализация

- `structlog` + `logging.handlers.RotatingFileHandler` (`logs/hodgkin.log`, 10 MB × 5).
- Middleware FastAPI: `request_id`, `session_id`, duration_ms.
- Опционально: audit-лог отдельным файлом `logs/audit.jsonl` для воспроизведения решений (маркировка до/после шага).

### 7.4. Конфигурация

```python
# config.py
LOG_LEVEL = "INFO"
LOG_DIR = "logs"
LOG_JSON = True
```

## 10. Критерии приёмки

- [ ] Логика расчёта совпадает с таблицей docx (ручная проверка ≥5 кейсов).
- [ ] Нет разветвлённого if-else по симптомам вне `petri/engine.py` и JSON.
- [ ] Навигация вперёд/назад с сохранением ответов.
- [ ] Результат при 0/0 отображается корректно.
- [ ] Логи пишутся на каждый шаг и финальный диагноз.
- [ ] Предупреждение при невыполнении условий перехода (без блокировки).
- [ ] Инструментальный блок: предсказание с учётом правила «хотя бы один признак».

## 11. Риски и уточнения на реализации

1. **Точные guard-условия** для b9/b10, b11/b12, b26/b27, b42/b43 — сверить с графом сети в docx (если есть отдельная схема); при расхождении — уточнить у автора модели.
2. **b19/b20 (пол)** — взаимоисключающие; в UI — radio «мужчина / женщина / не указано».
3. **Возраст b17 vs b24** — отдельные чекбоксы по критериям из таблицы, не свободный ввод возраста.
