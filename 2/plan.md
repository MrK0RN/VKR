# План реализации: дифференциальная диагностика ХАТ (болезнь Хашимото)

**Заболевание:** хронический аутоиммунный тиреоидит (ХАТ) — гипертрофическая и атрофическая формы  
**Исходная модель:** `Сеть Петри болезнь Хашимото.pdf` (таблица 1, рис. 1)  
**Стек:** Python 3.11+, FastAPI, Jinja2/HTML, session-based storage  
**Принцип:** программная интерпретация сети Петри (не if-else дерево), логика модели не меняется

> **Важно:** это **отдельный проект** от лимфомы Ходжкина (папка `1/`). Кодовая база, JSON-сеть и UI — независимые.

## 2. Модель сети Петри (каноническое описание)

### 2.1. Структура (по PDF)

```
Жалобы (b1–b9) → [b10, b11] → Осмотр (b14–b20) → [b12, b13] →
Лаборатория (b23–b34) → [b21, b22] → Инструментальные (b37–b40) → [b35, b36] →
[b41 гипертрофический | b42 атрофический]
```

- **Places:** признаки + счётчики веток `score_hyper`, `score_atrophic`.
- **Transitions:** между разделами; срабатывание описано в JSON, исполняется `PetriEngine`.

### 2.2. Веса

| Обозначение в таблице | Смысл в коде |
|----------------------|--------------|
| `(1)` в колонке гипертрофический | +1 к `score_hyper` |
| `(2)` в колонке атрофический | +2 к `score_atrophic` |
| `(1) / (1)` | +1 к обеим формам при одной отметке |
| `(1) / (2)` | +1 hyper, +2 atrophic |
| Пустая колонка | признак только для одной формы |

Примеры **общих** жалоб: b4–b9 — двойные веса; b5–b8 с разными весами по формам.

### 2.3. Промежуточные позиции (вариант А)

- **Не блокируют** переход к следующему разделу.
- Фиксируют выполнение условий ветвления; при невыполнении — **информационное предупреждение**, пользователь идёт дальше.
- Токены в b10/b11, b12/b13 и т.д. ставятся движком при `fire_transition`, не вручную в роутере.

### 2.4. Правило последнего блока (инструментальные b37–b40)

Аналогично проекту лимфомы:

- На этапах 1–3 переходы оцениваются по **накопленной сумме** весов по ветке (как в сети).
- На этапе 4: для «полного» срабатывания перехода к финалу по инструментальному блоку достаточно **≥1 отмеченного** признака b37–b40 с любым весом.
- Пустой инструментальный блок → баллы только с предыдущих этапов, результат всё равно показывается.

### 2.5. Итог

- `score_hyper`, `score_atrophic` — суммы по дугам.
- Победитель: `argmax`; при равенстве — явная формулировка.
- Опционально: explain по отмеченным places.

## 5. Движок сети Петри

### 5.1. Требования к «хорошей» реализации

1. **Единая точка истины** — все признаки и веса только в JSON.
2. **Разделение данных и исполнения** — `PetriNet` (структура) vs `PetriEngine` (динамика).
3. **Воспроизводимость** — из лога audit можно восстановить marking после каждого шага.
4. **Идемпотентность шага** — повторный POST того же раздела пересчитывает marking с нуля по сохранённым answers (избегает двойного начисления).
5. **Типизированные guards** — enum `FireCondition`: `SUM_GT_ZERO`, `ANY_ONE_MARKED`, `ALWAYS` (для финала без инструментальных).

### 5.2. Псевдокод `apply_section`

```python
def apply_section(engine: PetriEngine, answers: dict[str, bool]) -> SectionResult:
    engine.reset_section_scores_accumulator()  # или full recompute from all answers
    for place_id, marked in answers.items():
        if marked:
            engine.mark(place_id)  # adds tokens + propagates arc weights to scores
    tr = engine.evaluate_transition_for_section(engine.current_section)
    return SectionResult(
        scores=engine.scores.copy(),
        transition_fired=tr.fired,
        warning=tr.warning_message,
        intermediate_marked=tr.output_places,
    )
```

### 5.3. Тест-кейсы (обязательные)

| Кейс | Ожидание |
|------|----------|
| Только b1 | hyper=2, atrophic=0 |
| b4 отмечен | hyper=1, atrophic=1 |
| b5 отмечен | hyper=1, atrophic=2 |
| Только b20 | hyper=0, atrophic=2 |
| Пустой опрос | 0/0, winner=None |
| Только b39 | atrophic≥1, срабатывает any_one instrumental |
| b16+b17 не отмечены | без штрафа, пол опционален |

## 6. Веб-интерфейс

### 6.1. Четыре шага

1. **Жалобы** — b1–b9  
2. **Осмотр** — b14–b20 (+ предупреждение по b10/b11)  
3. **Лаборатория** — b23–b34 (две визуальные группы: «гипертрофическая» / «атрофическая» лаборатория, без смешивания логики в коде — просто группировка в UI)  
4. **Инструментальные** — b37–b40  

### 6.2. Навигация

- Вперёд / Назад с подгрузкой `answers` из сессии.
- На последнем шаге — кнопка «Получить результат» → `finalize()`.

### 6.3. Страница результата

```
Наиболее вероятная форма: Гипертрофический ХАТ
Баллы: Гипертрофический — 12 | Атрофический — 5

[Опционально] Признаки, повысившие вероятность гипертрофической формы:
  • b1 Ощущение «кома» в шее (+2)
  • b14 Увеличение шеи (+2)
  ...
```

## 7. Логирование

### 7.1. Цели

- Отладка расхождений с бумажной моделью.
- Аудит решений support-системы без хранения ПДн.

### 7.2. Стек

- `structlog` + стандартный `logging`.
- Файлы: `logs/hashimoto.log` (ротация), `logs/hashimoto_audit.jsonl` (по одной JSON-строке на шаг).

### 7.3. Схема audit-записи

```json
{
  "ts": "2026-05-15T15:00:00Z",
  "event": "section_submitted",
  "session_id": "abc123",
  "section": "complaints",
  "marked_places": ["b1", "b4"],
  "scores_after": { "hyper": 3, "atrophic": 1 },
  "transition": { "id": "t_complaints_to_exam", "fired": true },
  "warning_shown": null
}
```

### 7.4. События (как в проекте 1)

`session_started`, `place_marked`, `transition_evaluated`, `transition_warning_shown`, `diagnosis_completed`, `engine_error`.

### 7.5. Middleware

- `X-Request-ID` / генерация UUID на запрос.
- Логирование времени обработки POST `/step/{n}`.

## 11. Критерии приёмки

- [ ] Веса b1–b40, b41/b42 соответствуют таблице 1 PDF.
- [ ] Расчёт через PetriEngine, не через разрозненные if по симптомам.
- [ ] 0/0 отображается без ошибки.
- [ ] Предупреждение на этапе без «подозрений», переход не блокируется.

- [ ] Сессия сохраняет ответы при Back.
- [ ] Логи + audit.jsonl на каждый submit.
